# 前台联调与真机回归清单

这份清单是给当前微信商城小程序前台补强阶段用的，重点不是测真实支付、真实登录、真实数据库，而是把页面状态、主链路、展示细节和真机体验先收口。

当前说明：

- 前台第一轮联调和真机回归已经完成
- 这份文档现在主要作为后续复测参考
- 它不是当前最优先的剩余任务清单

建议把它当成一份“照着做”的操作手册：

1. 先确认自己现在测的是 `mock` 还是 `api`
2. 再按主链路走一遍
3. 再补页面状态和展示细节
4. 最后做一轮真机体验检查

## 先看懂：`mock` 和 `api` 到底有什么区别

当前项目的数据源开关在 `miniprogram/config/env.js`：

```js
mallDataSource: "mock"
```

这里的值决定前端数据从哪里来。

| 模式 | 数据从哪里来 | 要不要启动 `server/` | 适合测什么 | 不适合测什么 |
| --- | --- | --- | --- | --- |
| `mock` | 小程序本地 mock 服务 | 不需要 | 页面结构、空状态、错误态、文案、按钮状态、主链路演示 | 前后端接口联调、真实网络异常 |
| `api` | `apiBaseUrl` 指向的 HTTP 接口 | 需要 | 接口联调、字段映射、接口失败提示、提交流程 | 真正的生产环境能力，当前也还不是真实数据库 |

可以把它理解成：

- `mock`：前端自己带了一套“假数据 + 假流程”，最适合先把页面体验打磨稳定。
- `api`：前端通过 `wx.request` 去请求本地服务，最适合检查“前端和接口有没有对上”。

## 一个最容易搞混的点

`mock` 和 `api` 不是在微信开发者工具里点一个开关切换的。

真正控制模式的是项目文件：

- `miniprogram/config/env.js`

微信开发者工具里你通常只需要做两件事：

- 重新编译，让配置生效
- 在 `api` 模式下，按需要关闭“校验合法域名”之类的本地限制

所以你前面问“是不是去微信开发工具那里修改 api 或 mock”，更准确地说是：

- 不是在开发者工具里切模式
- 是改代码里的 `mallDataSource`
- 改完以后回到开发者工具重新编译

## 如何切到 `mock` 模式

### 步骤

1. 打开 `miniprogram/config/env.js`
2. 确认：

```js
mallDataSource: "mock"
```

3. 保存文件
4. 回到微信开发者工具，点击“编译”

### 你会看到什么

- 页面不依赖本地 `server/`
- 就算本地服务没启动，页面也还能正常打开
- 购物车、下单、订单详情这些流程仍然可以跑
- 开发者工具的 Network 面板里，不会出现你预期中的 `/api/...` 请求链路

### 适合什么时候用

- 先验页面结构和主链路有没有断
- 先查空状态、错误提示、按钮 loading、禁用态是否合理
- 演示时想要一个更稳定的环境

## 如何切到 `api` 模式

### 步骤

1. 打开 `miniprogram/config/env.js`
2. 把配置改成：

```js
mallDataSource: "api",
apiBaseUrl: "http://127.0.0.1:3000"
```

3. 启动本地服务：

```bash
cd server
npm install
npm run dev
```

4. 确认健康检查通过：

```bash
curl http://127.0.0.1:3000/health
```

预期应该能看到成功返回。

5. 回到微信开发者工具，重新编译
6. 如果开发者工具拦截本地请求，到“详情 -> 本地设置”里关闭本地调试相关校验项

### 你会看到什么

- 页面会发出 `/api/...` 请求
- 如果本地服务停掉，页面会出现请求失败、加载失败或者错误提示
- 接口字段不对时，更容易暴露真实联调问题

### 适合什么时候用

- 检查前端和接口字段是否对齐
- 检查提交流程是否真的经过接口
- 检查接口失败时前端提示是否自然

## 怎么确认自己现在测的真的是想要的模式

这是最实用的一节，尤其适合刚开始联调时自检。

### 判断自己是不是在 `mock` 模式

满足下面大多数现象，基本就是 `mock`：

1. `env.js` 里写的是 `mallDataSource: "mock"`
2. 本地 `server/` 没启动，首页和主链路还是能跑
3. 购物车、下单、订单详情还能继续演示
4. 你故意把本地服务停掉，页面也没有立刻全部报错

### 判断自己是不是在 `api` 模式

满足下面大多数现象，基本就是 `api`：

1. `env.js` 里写的是 `mallDataSource: "api"`
2. 本地服务启动后，`/health` 可以访问
3. Network 里能看到请求发到 `http://127.0.0.1:3000/api/...`
4. 你把本地服务停掉以后，刷新页面会报网络错误或加载失败

## 真机联调前，必须先知道的一个坑

当前配置里的：

```js
apiBaseUrl: "http://127.0.0.1:3000"
```

在微信开发者工具里联调本机服务通常没有问题，但到了手机真机上，`127.0.0.1` 指向的是“手机自己”，不是你的电脑。

这意味着：

- 开发者工具里 `api` 模式能通
- 真机预览时同样配置很可能直接请求失败

如果你想做“真机 + api 模式”联调，通常要改成电脑的局域网 IP，例如：

```js
apiBaseUrl: "http://192.168.1.23:3000"
```

同时确认：

1. 手机和电脑在同一个 Wi-Fi
2. 电脑服务已经启动
3. 手机浏览器能打开 `http://你的局域网IP:3000/health`
4. 微信开发者工具里的本地调试限制已经处理好

如果你这轮主要是查页面样式、空状态、长列表滚动、按钮布局，那真机阶段更建议优先用 `mock` 模式，稳定很多。

## 回归前准备

每一轮开始前，先做下面这 8 步。

1. 明确本轮目标
   - 查页面样式和流程稳定性，优先用 `mock`
   - 查接口联调和错误提示，优先用 `api`
2. 确认 `miniprogram/config/env.js` 当前值
3. 如果是 `api` 模式，先启动 `server/`
4. 用 `/health` 确认服务正常
5. 回到微信开发者工具重新编译
6. 清一次 Storage 或者至少确认购物车、地址、优惠券、订单是不是你预期的数据状态
7. 从首页重新进入，不要停留在旧页面缓存里直接测
8. 开始记录本轮结果：通过 / 可接受问题 / 阻塞问题

## 建议的回归顺序

建议不要一上来就全量乱点，按这个顺序最省时间。

1. `mock` 模式跑主链路
2. `mock` 模式补页面状态和展示细节
3. `api` 模式跑主链路
4. `api` 模式专查提交、状态变更、错误提示
5. 真机补滚动、底部安全区、文案换行、按钮布局

## 主链路详细回归

主链路是：

首页 -> 商品详情 -> 加购 -> 购物车 -> 结算 -> 下单 -> 订单列表 -> 订单详情

每一步都建议同时看“操作是否成功”和“页面反馈是否自然”。

| 环节 | 要做的操作 | 通过标准 | 常见失败信号 |
| --- | --- | --- | --- |
| 首页 | 打开首页，等待首屏加载完成 | banner、分销卡、推荐商品都能展示；空状态和错误态文案正常 | 骨架不消失、模块错位、按钮点击没反应 |
| 首页 -> 详情 | 点击商品卡进入商品详情 | 页面跳转成功；商品标题、价格、规格、卖点能显示 | 跳转失败、参数丢失、页面报“商品不存在” |
| 商品详情 | 切规格、改数量、看权益和图文区 | 规格切换能联动；数量不低于 `1` 不高于 `99`；当前选择摘要正确 | 规格切换后价格/文案不更新；数量越界 |
| 加入购物车 | 点击“加入购物车” | 按钮进入 loading；重复点不会重复提交；成功后数量刷新 | 一直 loading、不提示成功、重复加入多次 |
| 购物车 | 进入购物车，增减数量、删除商品、勾选商品 | 商品数量、价格、小计、合计都正确；空购物车时有明确空状态 | 金额不更新、删除后残留旧数据、空状态不友好 |
| 去结算 | 点击“去结算” | 能顺利进入结算页 | 按钮不可点却没有原因；点击后无反馈 |
| 结算页 | 检查地址、商品、优惠券、备注、应付金额 | 缺地址时有提示；空购物车时有提示；金额计算稳定 | 地址区空白、优惠券选择异常、金额前后不一致 |
| 提交订单 | 点击“提交订单” | 按钮有 loading；重复点击被拦截；成功后进入结果页或订单页 | 重复提交、失败无提示、成功后跳错页面 |
| 订单列表 | 查看各个 tab | 全部/待发货/待收货/已完成切换正常；空列表文案清楚 | tab 状态混乱、订单卡缺字段、状态文案不统一 |
| 订单详情 | 打开订单详情 | 能看到订单进度、商品清单、金额明细、收货信息、操作按钮 | 订单状态和按钮不匹配；金额明细缺项；空白块过多 |

## 订单与售后专项回归

这部分建议至少拿 3 种订单状态各测一次：待发货、待收货、已完成。

| 场景 | 要做的操作 | 通过标准 |
| --- | --- | --- |
| 待发货订单 | 进入订单详情 | 能看到待发货状态、取消入口、正确提示文案 |
| 待收货订单 | 进入订单详情后确认收货 | 确认前有二次确认；确认后状态更新自然 |
| 已完成订单 | 查看详情 | 可见“申请售后”入口，文案明确 |
| 已有售后记录 | 再次进入订单详情 | 不应继续显示“申请售后”；应该显示“查看售后” |
| 发起售后 | 进入售后页并提交 | 原因选择、说明输入、提交 loading、成功反馈都正常 |
| 重复提交售后 | 连续点击提交 | 重复点击被拦截，不会重复创建记录 |
| 不可售后场景 | 打开不支持售后的订单 | 页面有明确原因提示，不是只把按钮静默隐藏 |

## 页面状态专项检查

这一轮不是看业务逻辑，而是专门看 `loading / error / empty / notFound` 是否都像一个完整产品。

建议逐页过下面这些页面：

- 首页
- 分类页
- 搜索页
- 商品详情页
- 购物车页
- 结算页
- 订单列表页
- 订单详情页
- 支付结果页
- 售后申请页
- 个人中心
- 地址列表页
- 地址编辑页
- 领券中心
- 分销中心
- 团队页
- 佣金明细页
- 海报页
- 授权页

每个页面至少看这 4 件事：

1. `loading` 时有没有骨架、占位或明确的加载提示
2. `error` 时有没有人能看懂的报错文案和重试入口
3. `empty` 时有没有解释当前为什么为空，以及下一步该去哪
4. `notFound` 或参数缺失时，页面是不是能优雅兜底，而不是白屏

## 展示细节专项检查

这轮主要是查“能不能演示得更像一个完成度高的产品”。

| 页面 | 重点检查项 |
| --- | --- |
| 首页 | banner 比例、分销卡信息、商品卖点、价格层级、按钮文案 |
| 商品详情 | 指标卡、权益说明、图文详情、当前选择摘要、底部操作栏 |
| 订单列表 | 状态标签、金额层级、按钮排列、空状态插画和文案 |
| 订单详情 | 流程进度、金额明细、收货信息、温馨提示、售后按钮表达 |
| 售后页 | 原因选择、说明输入区、处理说明、提交按钮状态 |
| 分销中心 | 等级信息、进度条、动作卡片、新手引导文案 |
| 个人中心 | 订单入口、领券入口、分销入口、地址入口是否统一易懂 |
| 海报页 | 标题、券面、二维码示意、按钮区、长文案换行 |

## 真机重点确认

真机检查和开发者工具检查不是一回事，真机阶段更容易暴露样式和交互问题。

至少确认下面这些点：

1. 长页面滚动是否流畅，有没有明显掉帧
2. 底部按钮是否被安全区遮挡
3. 多按钮并排时，在窄屏下会不会挤压或换行难看
4. 标题、副标题、价格、标签在小屏上有没有截断或重叠
5. toast、弹窗、loading 的出现时机是否自然
6. 页面返回上一层后，数据和按钮状态是否正确恢复
7. 连续点击、快速返回、重复进入页面时，会不会出现脏状态

## 建议怎么记录结果

每回归一轮，建议至少记 3 类结果。

| 结果类型 | 含义 | 例子 |
| --- | --- | --- |
| 通过 | 页面和链路符合预期，可以继续下一轮 | 首页、详情、加购、结算都跑通 |
| 可接受问题 | 不阻塞演示，但建议后续优化 | 某个空状态文案偏生硬，按钮间距还可再调 |
| 阻塞问题 | 会影响主链路或让用户无法理解页面 | 下单按钮无响应、订单详情打不开、真机底部按钮被遮挡 |

## 一轮标准回归的最小执行版本

如果今天时间不多，至少完成下面这一版：

1. 用 `mock` 模式跑一遍主链路
2. 用 `api` 模式再跑一遍主链路
3. 检查订单详情和售后入口
4. 真机看首页、商品详情、订单详情 3 个长页面
5. 记录本轮的通过项、可接受问题、阻塞问题

做到这一步，就已经是一轮比较合格的前台联调与回归了。
