# 正式上线配套注意事项

更新时间：2026-04-08

这份文档专门面向你们现在这个真实状态：

- 后端已经部署在微信云托管
- MySQL 已经在云上并完成过联调
- 小程序前后端主链路已基本跑通
- 当前真正没完成的是：
  - 真实微信登录的云上配置与真机验收
  - 微信支付
  - 正式上线前的配套收口

所以这份文档不再把重点放在“怎么上云”，而是回答：

- 现在离正式上线还差哪些硬前置
- 哪些事情是代码问题，哪些是配套问题
- 应该按什么顺序推进，才不容易返工

## 一句话结论

如果按“能正式对外卖货”来理解，现在最大的阻塞已经不是云托管和数据库，而是：

1. 主体 / 小程序账号 / 备案 / 类目
2. 真实微信登录
3. 微信支付与订单支付闭环
4. 运营、客服、售后、监控、管理员、安全这些上线配套

更直白一点说：

- 现在这套系统已经接近“可上线工程”
- 但还不能算“可正式营业的线上商城”
- 真正卡上线节奏的，是资质、支付和运营配套，不是页面数量

## 现在最容易误判的地方

下面这些“已经通了”的事情，不等于“已经可正式上线”：

- 云托管服务能访问
- 云上 MySQL 能落库
- 小程序真机能打开页面
- 商品、购物车、下单主链路能跑
- 本地和默认测试通过

这些只能说明：

- 工程基础盘已经稳了
- 云上运行路径已经具备

但不能说明：

- 用户能真实完成微信账号登录
- 用户能真实完成付款
- 你们能在正式环境稳定处理订单、售后、退款、异常单
- 小程序已经具备提审和正式对外发布条件

## 上线前的硬阻塞

下面这些如果没完成，就不应该把项目判断成“正式可营业”。

### 1. 主体、小程序账号、备案、类目

至少要确认：

- 正式运营主体已经定下来
- 正式小程序账号已经注册并认证
- 小程序备案已经按当前后台要求完成
- 小程序服务类目和实际经营范围一致
- 若类目涉及额外资质，材料已经准备好

这块更详细的资料准备，优先看：

- [entity-go-live-checklist.md](/Users/tongqianqiu/store/wechat-mini-shop/docs/current/entity-go-live-checklist.md)

### 2. 真实微信登录

当前仓库里已经有两段真实链路：

- `wx.login -> /api/auth/session -> code2Session`，负责建立真实微信会话
- `getPhoneNumber -> /api/auth/authorize -> getuserphonenumber`，负责拿真实手机号并落库

这次也补上了生产防护和可观测性，但“代码存在”仍然不等于“正式可用”。

当前仓库已补到这里：

- 服务端可通过 `GET /api/auth/login-readiness` 自检真实微信登录配置是否就绪
- `production` 环境默认不再允许 `mock_wechat`
- 可通过 `ALLOW_MOCK_WECHAT_LOGIN=false` 显式关闭 mock 登录，避免演示会话进入正式环境
- 登录页已经调整为生产链路：先显式触发 `wx.login` 建会话，再通过手机号按钮完成微信手机号授权

正式上线前，至少要确认：

- 小程序使用的真的是正式 `AppID`
- 服务端已经配置正式 `WECHAT_APP_ID`
- 服务端已经配置正式 `WECHAT_APP_SECRET`
- 小程序端 `sessionLoginMode` 保持为 `wechat`
- 真机首登、重登、退出重登都走真实微信登录，不再依赖 `mock`
- 真机点击手机号授权后，服务端能收到 `/api/auth/authorize`，并把真实手机号写回当前用户
- 邀请绑定、分享入参、会话续期在真实登录下也能正常工作
- `GET /api/auth/login-readiness` 返回 `wechatMiniProgram.configured=true`
- `GET /api/auth/login-readiness` 返回 `mockWechat.enabled=false`

如果这一块没打通：

- 当前系统最多算“云上联调可用”
- 还不能算“正式线上可用”

### 3. 微信支付

如果你们的首发形态还是正常电商交易，那么微信支付不是“可选增强项”，而是硬阻塞。

至少要补齐：

- 微信支付商户号
- 小程序与商户号绑定关系
- 支付所需密钥、证书或平台证书管理方案
- 服务端统一下单 / 预支付参数生成
- 小程序端 `wx.requestPayment`
- 支付结果通知回调
- 支付结果回查
- 订单与支付结果的幂等处理

如果只有支付结果页、订单状态页，但没有真实支付回调和回查：

- 用户侧会看到“像能支付”
- 但系统侧并没有真正的支付闭环

## 按当前状态最推荐的推进顺序

不要并行到处开坑，当前最稳的顺序是：

1. 先把正式环境变量和正式小程序资产收口
2. 再打通真实微信登录
3. 然后设计并接入微信支付闭环
4. 接着补上线运营配套
5. 最后做提审前回归、提审、灰度和发布

这样排的原因很简单：

- 你们云上部署已经不是主阻塞
- 真实登录和支付是后续所有回归的基础
- 如果支付先做、登录没稳，后面排查问题会很乱

## 上线配套检查清单

下面这部分是这次最核心的“照着做”清单。

### A. 正式配置收口

必须确认：

- 云托管环境变量已经和正式环境一一对应
- `STOREFRONT_DATA_SOURCE=prisma`
- `NODE_ENV=production`
- `ALLOW_MOCK_WECHAT_LOGIN=false`
- `ADMIN_USERS` 已使用正式强密码生成，不再使用演示账号
- `DATABASE_URL` 指向正式云库
- 如果后台会被浏览器独立访问，`CORS_ORIGINS` 已按实际来源配置

小程序端必须确认：

- `mallDataSource="api"`
- `sessionLoginMode="wechat"`
- `requestTransport="cloud"`
- `cloud.env`、`cloud.service`、`cloud.path` 都与当前线上环境一致

当前对应位置：

- [env.js](/Users/tongqianqiu/store/wechat-mini-shop/miniprogram/config/env.js)
- [environment-variables.md](/Users/tongqianqiu/store/wechat-mini-shop/docs/setup/environment-variables.md)

### B. 真实微信登录收口

当前后端代码已经具备真实微信登录链路，这一节的重点不再是“从零开发”，而是“把正式环境收严并完成真机验收”。

上线前至少做完这一轮真机回归：

- 新用户首次进入小程序
- 已有用户再次进入小程序
- 清空本地缓存后重新进入
- 登录态失效后自动补会话
- 从登录页点击“微信一键登录”后，能成功建立真实会话
- 从登录页点击“手机号一键授权”后，能成功拿到真实手机号
- 分享带 `inviterUserId` 的进入路径
- 异常场景：`wx.login code` 失效、手机号授权 `code` 失效、服务端配置缺失、用户取消手机号授权、网络失败
- 部署后先访问 `GET /api/auth/login-readiness`，确认 `wechatMiniProgram.configured=true`
- 部署后再确认 `mockWechat.enabled=false`
- 部署后真机确认一次 `/api/auth/session` 和 `/api/auth/authorize` 都有真实请求日志

至少要有一个明确结论：

- 当前线上环境已经完全不依赖 `mock session`
- 当前线上用户看到的“登录动作”是手机号授权按钮，`wx.login` 本身是静默会话建立，不会像传统账号密码登录那样单独弹一页
- 线上真实微信登录故障时，能从云托管日志按 `requestId` 快速定位

### C. 微信支付与订单闭环

支付不是只把 `wx.requestPayment` 调起来就算完，真正要收口的是“订单 + 支付 + 回调 + 回查”。

至少要确认：

- 下单后有明确的“待支付”订单状态
- 服务端生成支付参数前，会校验订单金额和订单状态
- 小程序端唤起支付后，不能把前端结果当作唯一成功依据
- 服务端支付回调成功后，才把订单推进到“已支付 / 待发货”
- 对同一笔回调做幂等处理，避免重复改状态
- 有支付结果回查能力，避免回调丢失时订单悬挂
- 超时未支付订单的关闭策略已经明确

这块最容易漏掉的不是“拉起支付”，而是：

- 回调签名校验
- 幂等
- 金额校验
- 订单号和支付单号映射
- 支付成功但前端没收到结果时的补偿逻辑

### D. 售后、退款、财务口径

当前项目已经有售后申请和后台审核，但正式上线前要把口径说清楚：

- 首发是否支持原路退款
- 首发是否只支持人工退款
- 售后通过后，谁负责执行退款
- 后台订单状态和退款状态如何记录
- 分销佣金遇到退款时是否冲回

如果首发阶段来不及做完整自动退款，也至少要有一份人工 SOP：

- 谁来判断退款
- 在哪里操作退款
- 如何同步订单状态
- 如何留痕

### E. 运营后台与账号安全

至少要确认：

- 正式管理员账号已经配置完成
- 演示账号已不再用于生产
- 账号权限按角色拆分，不要所有人共用超管
- 后台登录、发货、售后审核的责任人已经确定
- 后台访问方式已经明确，不要把调试入口直接当生产入口长期裸露

当前后台和账号相关代码位置：

- [auth.js](/Users/tongqianqiu/store/wechat-mini-shop/server/src/admin/auth.js)
- [router.js](/Users/tongqianqiu/store/wechat-mini-shop/server/src/admin/router.js)

### F. 监控、日志、备份、应急

这部分最容易被忽略，但上线后最救命。

至少要确认：

- 云托管日志可以快速定位请求失败
- 关键接口错误能按 `requestId` 追踪
- 数据库已经有自动备份策略
- 至少演练过一次“如果误删数据 / 订单异常，怎么恢复”
- 发布时有人值守观察支付、下单、发货、售后相关日志
- 有明确回滚方案：回滚代码、回滚配置、暂停提审或暂停交易

### G. 提审与用户侧合规信息

正式提审前，至少要逐项确认：

- 小程序名称、简介、类目、截图、体验路径已准备
- 用户能看到必要的客服与售后信息
- 用户协议、隐私政策、售后说明、发货说明等信息已经明确
- 如果当前公众平台后台要求补充隐私保护配置，已经按正式版本配置完成

这块不要死记旧资料：

- 以当前微信公众平台 / 小程序后台 / 微信支付后台的实时要求为准
- 提审前一天最好再按后台实际项核一遍

## 首发最少要跑通的真实业务链

如果按“第一版正式上线”理解，建议至少把下面这几条线上真链路全部走完：

1. 新用户进入小程序并完成真实微信登录与手机号授权
2. 首页 -> 商品详情 -> 加购 -> 购物车 -> 结算
3. 提交订单 -> 微信支付 -> 支付成功 -> 订单进入待发货
4. 后台查看订单 -> 发货 -> 小程序端看到待收货
5. 用户确认收货
6. 发起售后 -> 后台审核 -> 状态同步

如果这 6 条里有任何一条还只能靠 mock 或手工补状态，那都不建议直接对外发版。

## 当前最建议优先补的 3 件事

如果接下来只做三件事，我建议就是：

1. 把正式小程序资产、正式环境变量和正式管理员账号彻底收口
2. 用真机把真实微信登录和手机号授权完整打通
3. 先设计清楚微信支付的订单状态机、回调、回查、幂等，再开始接代码

原因：

- 现在已经不是缺基础设施
- 现在最怕的是“看起来快上线了，但核心交易链没闭环”
- 先把登录和支付设计收严，后面回归和提审会顺很多

## 明确的上线判断线

可以把当前项目的判断线收成一句话：

- 云托管和云库已经通，说明“工程部署”基本过关
- 真实微信登录和微信支付没通，说明“正式营业能力”还没过关

所以当前最合理的目标，不是继续扩页面，而是：

- 先把正式登录和支付闭环做实
- 再把监控、管理员、售后、提审资料这些上线配套补齐

这才叫真正的“上线收口”。
