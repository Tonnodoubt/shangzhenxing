# 仓库目录命名说明

更新时间：2026-04-08

这份只回答一件事：

- 现在仓库里几个大文件夹的名字，算不算标准

## 一句话结论

当前主目录名整体是合格的。

最主要的 3 个业务目录：

- `docs/`
- `miniprogram/`
- `server/`

这 3 个名字都清楚、常见、能长期用，当前不建议为了“更标准”去改。

## 现在哪些名字可以继续保留

### 仓库根目录

- `wechat-mini-shop`

这个名字没有问题。

它表达的是：

- 微信小程序商城项目

如果后面不会扩成多端大平台，这个名字完全可以继续用。

### 主业务目录

- `docs/`
- `miniprogram/`
- `server/`

这 3 个目录是当前仓库里最标准、最清楚的一组。

简单理解：

- `docs/` 放文档
- `miniprogram/` 放小程序前端
- `server/` 放后端

这组命名现在不用动。

## 现在风格不完全统一，但问题不大的是哪块

仓库外面的归档目录：

- `../archived-docs/`

它本身不是错，但和仓库里的目录风格不完全一致。

因为仓库里主要是：

- 短英文目录名

而它是：

- 英文复合归档名

所以它的问题不是“不能用”，而是“风格略不统一”。

如果以后真想统一风格，可以考虑改成：

- `docs-archive`

但这不是当前优先级。

## 当前最不标准的，其实不是大目录，而是 `shared`

现在仓库里有多套 `shared`：

- `shared/`
- `server/shared/`
- `server/src/shared/`
- `miniprogram/shared/`

这才是当前命名上最容易让人绕的地方。

## 为什么 `shared` 容易绕

因为它们名字很像，但职责并不完全一样。

当前更接近下面这种理解：

- `shared/`
  更像跨端共用源文件

- `server/shared/`
  更像服务端运行时副本

- `miniprogram/shared/`
  更像小程序运行时副本

- `server/src/shared/`
  更像服务端内部公共模块

问题就在这里：

- 名字都叫 `shared`
- 但语义并不在同一层

所以新同学一进来，很容易分不清“到底该改哪一份”。

## 当前最稳的做法

现在不建议直接重命名这些目录。

原因很简单：

- 这里已经牵到真实引用路径
- 现在直接改名，容易把运行链路改炸
- 当前优先级也不是目录美化，而是主链路稳定和文档收口

## 如果以后要整理，建议顺序

建议顺序只记这 3 条：

1. 先保留 `wechat-mini-shop / docs / miniprogram / server`
2. `archived-docs` 以后再决定要不要统一成 `docs-archive`
3. 真要做目录命名整理，第一优先级先处理 `shared` 体系，而不是主目录

## 以后如果真要改 `shared`，建议怎么改

不建议现在动。

但如果后面要正式收目录，建议至少做到一件事：

- 让“跨端共用目录”和“服务端内部公共目录”别再都叫 `shared`

更稳的方向是：

- 保留一个真正的跨端 `shared/`
- 把 `server/src/shared/` 这类服务端内部目录改成更明确的名字

比如以后可以再评估：

- `server/src/common/`
- `server/src/lib/`

先记住重点就够：

- 现在最需要保留的是主目录稳定
- 以后真要整理，先收 `shared`，不要先动主目录
