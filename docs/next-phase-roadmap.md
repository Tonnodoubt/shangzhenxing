如果你现在只想看“还剩什么没做、先做什么”，优先看：

这份文档是给当前微信商城小程序“下一阶段”准备的。

目标不是一下子把数据库、登录、支付、后台全做完，而是按依赖关系一层层推进，沿着微信云托管的真实部署路径往前走，同时继续保住现在已经跑通的前台演示环境。

补一条当前已完成的里程碑，避免和最新仓库状态错位：

- 本地 Docker MySQL 已补 `server/compose.local-mysql.yml`
- 本地 Prisma 商品种子已补 `npm run prisma:seed:storefront`

- 最小履约控制台已补 `/admin-console/`
- Prisma 真库回归已覆盖后台发货与售后审核

## 先记住一个总原则

下一阶段建议按下面顺序做：

1. 先把最小履约能力跑通并稳定住
2. 主体确定后再做小程序备案与真实微信登录
3. 准备腾讯云 MySQL 和微信云托管
4. 最后再做真实微信支付

不要一上来先做支付。

因为支付依赖：

- 真实订单落库
- 真实用户身份
- 稳定的登录态
- 可回查的支付结果

前面三件没打稳，支付通常会变成最难排查的一环。

## 登录态应该怎么接

下一阶段先接“登录态”，不要马上接“真实微信登录”。

可以理解成两步：

### 第一步：先跑通登录态

这一步先不碰微信开放能力，先把“会话机制”跑起来。

前端要做的事：

- 本地保存 `sessionToken`
- 请求时自动带 `Authorization`
- 接口返回 `401` 时，清掉本地会话并引导去授权页

后端要做的事：

- 新增 `user_sessions` 表
- 提供“创建会话 / 校验会话 / 获取当前用户 / 退出登录”能力
- 增加统一鉴权中间件

建议先补的接口：

- `POST /api/auth/session`
- `GET /api/me`
- `POST /api/auth/logout`

说明：

- 这一步的 `POST /api/auth/session` 可以先用“模拟登录”方式创建会话
- 目的不是假装已经完成微信登录
- 目的是先把“前端带 token -> 后端识别用户 -> 返回当前用户数据”这条链路打通

### 第二步：再替换成真实微信登录

等登录态本身跑稳以后，再把登录入口替换成：

1. 小程序端调用 `wx.login()`
2. 拿到 `code` 发给后端
3. 后端调用微信 `code2Session`
4. 用 `openid` 查找或创建用户
5. 后端签发自己的 `sessionToken`
6. 前端保存 token，后续请求继续复用

这样做的好处是：

- 出问题时你能判断是“登录态机制问题”还是“微信开放平台问题”
- 不会把两个难题混在同一天排查

## 真实数据库这一段应该怎么推进

当前不需要因为还没有云上 `DATABASE_URL` 就停住。

更稳的做法是：

1. 继续保留本地 Docker MySQL 作为真库联调底座
2. 本地需要回归时直接执行：
   `npm run prisma:migrate:deploy`
   `npm run prisma:seed:storefront`
   `npm run storefront:regression:prisma`
3. 再准备未来上线要用的云上 `DATABASE_URL`
4. 主体确定后补 `WECHAT_APP_ID / WECHAT_APP_SECRET`
5. 把 `sessionLoginMode` 从 `mock` 切到 `wechat`

这样做的好处是：

- 不会因为缺少云资源而让项目整体停摆
- 也不会偏离真实上线需要的结构

## 建议的实际实施顺序

如果你接下来准备一周一周往下做，最稳的顺序是：

1. 梳理生产环境变量并确认主体资料
2. 主体确定后注册小程序并提交备案
3. 补 `WECHAT_APP_ID / WECHAT_APP_SECRET`
4. 把 `sessionLoginMode` 改成 `wechat`
5. 准备腾讯云 MySQL 和微信云托管
6. 最后再做真实微信支付

## 给新手的落地建议

你现在最需要记住的一句话是：

- 不是“数据库还没到位，所以先别动”
- 而是“先把结构和登录态准备好，数据库连接串到了就直接切”

这会比“同时上数据库 + 微信登录 + 微信支付”轻松很多。

## 下一轮开始前的自检清单

开始真正切数据库前，先确认下面几件事：

- 我是不是已经接受“先不做支付”
- 我是不是已经把后端逻辑从前端 mock 逻辑里分开了

如果这四个问题都能回答“是”，下一阶段就会顺很多。
